## Problem

The `feature/cms-engine` branch has 25 commits of CMS engine work — all 9 content types migrated to the registry-driven operations layer — that have never landed on `develop`. Before that branch can merge, two things need to happen:

1. **A0:** Verify the engine's field definitions match the live database schema. The engine's `src/cms/types/*.ts` files were written against an assumed DB shape; if that assumption is wrong, A2 would ship broken migrations.
2. **A2:** Land the branch cleanly on top of A1's security architecture, resolve the remaining import cleanup, and confirm nothing was silently broken.

A0 is a gate for A2. A2 is a gate for A3 (generic admin form).

---

## Goals

**A0:**
- Confirm the live DB schema matches the CMS engine's field definitions across all 9 content types.
- Produce a committed artifact that records what was checked and what (if anything) was found.
- Gate A2 if mismatches exceed the threshold (> 3 ❌ + ⚠️ findings).

**A2:**
- Rebase `feature/cms-engine` cleanly onto post-A1 `develop`.
- Audit and fix `src/cms/operations/*.ts` for compliance with A1's ESLint module boundary.
- Eliminate the deprecated `@/lib/supabase` barrel (8 remaining importers).
- Delete the stale `src/types/supabase-new.ts` type snapshot.
- Verify trash/restore functionality is covered before deleting any old admin components.
- Merge to `develop`; all 9 content types on the engine.

## Non-goals

- Introducing any new features or content types.
- Changing the public rendering path. No user-visible change ships with A0 or A2.
- Setting up the generic admin form (A3).
- Migrating the block content model (Phase B).
- Fixing any pre-existing lint warnings (459 known; do not touch).

## Sequencing

**A0 can run now.** It only needs a live Supabase connection and the `feature/cms-engine` worktree.

**A2 must wait for A1 to merge first.** A1 restructures `src/lib/` (introduces `internal-queries.ts`, `public-query.ts`, ESLint boundary) in ways that will conflict with the cms-engine branch. Merging cms-engine before A1 would mean fixing conflicts twice and could introduce A1 ESLint violations immediately after merge.

## A0 — Schema Cross-Check

### Method

The project works directly against the live Supabase instance (no local DB). The authoritative schema record is the auto-generated `src/types/supabase.ts`.

**Step 1 — Regenerate types from live DB:**
```bash
supabase gen types typescript --project-id <project-ref> > src/types/supabase.ts
```
Commit the result if it differs from what is currently committed. If it differs significantly, that itself is a finding worth noting in the artifact.

**Step 2 — Check for unapplied migrations:**
```bash
supabase db diff
```
If any migrations in `supabase/migrations/` have not been applied to the live DB, stop and apply them before continuing. A0 is meaningless if the DB is behind the migration history.

**Step 3 — Compare engine definitions against generated types:**
For each of the 9 content types in `src/cms/types/*.ts` (in the `feature/cms-engine` worktree), manually compare the defined fields against the corresponding table in `src/types/supabase.ts`. Check:
- Every field defined in the engine type has a matching column in the DB (same name, compatible type).
- Every relationship defined in the engine type has a corresponding junction table in the DB.
- No fields defined in the engine type are missing from the DB (❌).
- No type mismatches between engine definition and DB column type (⚠️).
- Note any DB columns not referenced in the engine definition (~) — informational, not a blocker.

**Step 4 — Commit findings artifact:**
Save results to `docs/superpowers/research/2026-05-26-schema-crosscheck.md` and commit.

### Gate condition

If the artifact records > 3 ❌ (missing columns) + ⚠️ (type mismatches) findings across all types combined, **pause A2** and correct the engine type definitions in `src/cms/types/*.ts` before proceeding.

### Artifact format

```markdown
# Schema Cross-Check — 2026-05-26

**Supabase types regenerated:** yes / no (diff: N lines changed)
**Unapplied migrations:** none / list

## Results by content type

### algorithms
| Field | Engine type | DB column | Status |
|-------|------------|-----------|--------|
| title | text | text | ✅ |
| ... | ... | ... | ... |

### case-studies
...

## Summary
- ✅ Matches: N
- ❌ Missing from DB: N
- ⚠️ Type mismatches: N
- ~ Extra DB columns (informational): N

**Gate result:** PASS / FAIL (N mismatches, threshold 3)
```

## A2 — CMS Engine Integration

### Step 1 — Rebase onto post-A1 develop

```bash
cd .worktrees/cms-engine
git fetch origin
git rebase origin/develop
```

Expected conflict surface: `src/lib/` files, specifically anything the engine touches that A1 restructured (`supabase-server.ts`, `supabase-browser.ts`, any file A1 added to the module boundary allow-list).

Resolution rule: **keep A1's security architecture**. If a file conflicts between A1's version and the engine's version, preserve A1's module boundary, ESLint rules, and `internal-queries.ts` / `public-query.ts` imports. Adapt engine code to the A1 shape, not the other way around.

### Step 2 — Audit operations layer for A1 ESLint compliance

A1 introduces an ESLint `no-restricted-imports` rule: only files on the allow-list may import from `internal-queries.ts`. The engine's `src/cms/operations/*.ts` files make Supabase `.from()` calls and need to be checked.

For each file in `src/cms/operations/`:
- Does it import from `@/lib/supabase-server` or `@/lib/supabase-browser` directly? → Check whether it should go through `internal-queries.ts` instead.
- Does it make raw `.from()` calls that should be wrapped in `fromTable()`? → Update.
- Is it a legitimate admin/build-time path that belongs on the allow-list? → Add to the ESLint allow-list explicitly.

The outcome must be: `npm run lint` passes with 0 errors on the rebased branch.

### Step 3 — Update 8 importers off `@/lib/supabase`

These files import from the deprecated `@/lib/supabase` barrel:

| File | What it imports | Correct target |
|------|----------------|----------------|
| `src/app/auth/callback/route.ts` | `createServerSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/case-studies/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/audit-log/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/case-studies/trash/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/api/case-studies/permanent-delete/route.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/api/case-studies/restore/route.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/api/audit-log/export/route.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/utils/content-management.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |

After updating all 8, confirm `src/lib/supabase.ts` has zero importers remaining, then delete it.

### Step 4 — Delete stale type snapshot

Delete `src/types/supabase-new.ts`. It is an older snapshot of the auto-generated types; `src/types/supabase.ts` (refreshed in A0) is the current authoritative version. Search for any importers of `supabase-new` and update them to `supabase` before deleting.

### Step 5 — Verify trash/restore coverage

Before deleting any old admin components for case-studies, confirm these routes are covered by the engine or have been intentionally preserved outside it:
- `src/app/admin/case-studies/trash/` — trash listing
- `src/app/api/case-studies/restore/` — restore action
- `src/app/api/case-studies/permanent-delete/` — permanent delete action

These are soft-delete lifecycle routes that sit outside the standard CRUD operations layer. Do not delete them — they are not part of what the engine replaces. The engine covers create/update/publish/unpublish/soft-delete; these routes handle the trash queue and are kept as-is.

### Step 6 — Quality gates

```bash
npx vitest run          # all tests pass
npm run lint            # 0 errors (459 pre-existing warnings unchanged)
npx tsc --noEmit --skipLibCheck  # no new type errors
npx next build          # build succeeds
```

### Step 7 — Merge to develop

Open a PR from `feature/cms-engine` to `develop`. Review, then merge.
