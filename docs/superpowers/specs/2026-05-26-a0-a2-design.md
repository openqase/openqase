## Problem

The `feature/cms-engine` branch has 25 commits of CMS engine work — all 9 content types migrated to the registry-driven operations layer — that have never landed on `develop`. Before that branch can merge, two things need to happen:

1. **A0:** Verify the engine's field definitions match the live database schema. The engine's `src/cms/types/*.ts` files were written against an assumed DB shape; if that assumption is wrong, A2 would ship broken code.
2. **A2:** Land the branch cleanly on top of A1's security architecture, resolve the remaining import cleanup, and confirm nothing was silently broken.

A0 is a gate for A2. A2 is a gate for A3 (generic admin form).

---

## Goals

**A0:**
- Confirm the live DB schema matches the CMS engine's field definitions across all 9 content types.
- Produce a committed artifact that records what was checked and what (if anything) was found.

**A2:**
- Rebase `feature/cms-engine` cleanly onto post-A1 `develop`.

- Delete the stale `src/types/supabase-new.ts` type snapshot.
- Verify trash/restore/permanent-delete functionality works on the rebased branch.
- Merge to `develop`; all 9 content types on the engine.

## Non-goals

- Introducing any new features or content types.
- Changing the public rendering path. No user-visible change ships with A0 or A2.
- Setting up the generic admin form (A3).
- Migrating the block content model (Phase B).
- Fixing any pre-existing lint warnings (459 known; do not touch).

## Sequencing

**A0 runs on `develop` now.** It regenerates `src/types/supabase.ts` on `develop`, commits if changed, and runs the cross-check script against the engine worktree's type definitions.

**A2 must wait for A1 to merge first.** A1 restructures `src/lib/` (introduces `internal-queries.ts`, `public-query.ts`, ESLint boundary) in ways that will conflict with the cms-engine branch. Merging cms-engine before A1 would mean fixing conflicts twice and could introduce A1 ESLint violations immediately after merge.

**A0 result freshness:** If A1 slips and A2 is delayed more than ~2 weeks after A0 completes, re-run the cross-check script before starting A2. Schema can drift; the artifact has a shelf life.

## A0 — Schema Cross-Check

### Method

The project works directly against the live Supabase instance (no local DB). The authoritative schema record is the auto-generated `src/types/supabase.ts`.

**Step 1 — Regenerate types on `develop`:**

```bash
supabase gen types typescript --project-id <project-ref> > src/types/supabase.ts
```

Run this on the `develop` branch (not the engine worktree). Commit the result to `develop` if it differs from what is currently committed — note the diff size in the artifact. Then when the engine branch rebases onto `develop` in A2, it picks up the refreshed types automatically.

**Step 2 — Check for unapplied migrations:**

```bash
supabase db diff
```

If `supabase db diff` reports migrations that have not been applied to the live DB: print the unapplied migration file names to stderr, write them to the artifact's `Unapplied migrations` field, and exit with code 2. Do not apply them — applying migrations to the live DB is a separate risk decision requiring explicit confirmation outside this plan. Pause A0 and resolve the unapplied migrations before continuing. Code 2 is distinct from the gate-failure exit code (1) so the caller can distinguish the two failure modes.

**Step 3 — Run the cross-check script:**

Run `scripts/schema-crosscheck.ts` (see Script Design below) from the repo root. The script resolves engine type definitions from `.worktrees/cms-engine/src/cms/types/` by default (override with `--engine-types <path>`) and reads `src/types/supabase.ts` for the generated DB types. It writes the findings markdown to stdout:

```bash
npx tsx scripts/schema-crosscheck.ts > docs/superpowers/research/2026-05-26-schema-crosscheck.md
```

Commit the artifact. The script exits with code 1 if the gate condition fails, code 2 if unapplied migrations were detected in Step 2 and re-checked here, code 0 if PASS.

### Script Design

`scripts/schema-crosscheck.ts` — reusable, exits non-zero on gate failure.

**CLI:**
```
npx tsx scripts/schema-crosscheck.ts [--engine-types <path>]

  --engine-types  Path to engine type definitions directory.
                  Default: .worktrees/cms-engine/src/cms/types
```

The script always reads `src/types/supabase.ts` from the repo root (where it was regenerated in Step 1).

**What it does:**
1. Loads all `*.ts` files from the engine types directory; imports each `defineContentType()` export.
2. Reads `Database['public']['Tables']` from `src/types/supabase.ts` for column names and types, and `Database['public']['Tables'][T]['Relationships']` for FK metadata.
3. For each content type, compares every field name in the engine definition against the table's `Row` shape.
4. For each relationship, checks: (a) the `junction` table exists in `Database['public']['Tables']`; (b) the `foreignKey` and `targetKey` columns exist in that junction table's `Row` shape. Existence-only is insufficient — a junction table missing its FK columns is a structural mismatch.
5. Uses the generated `Relationships` array for FK detection rather than a name-suffix heuristic — more reliable for non-standard column names.
6. Outputs a markdown report to stdout; exits with code 1 on gate failure.

**Status categories:**
- `✅` — field name exists in DB with a compatible type
- `❌` — field defined in engine but missing from DB entirely (including FK/junction columns)
- `⚠️` — field exists but type is incompatible
- `~` — column exists in DB but not referenced in engine definition (informational, not a blocker)

**Type compatibility mapping** (engine field type → acceptable DB column types):

| Engine type | Acceptable DB types | Detection mechanism |
|-------------|-------------------|-------------------|
| `text`, `textarea`, `markdown`, `slug` | `text`, `character varying` | field type |
| `boolean` | `boolean` | field type |
| `number` | `integer`, `bigint`, `numeric`, `smallint` | field type |
| `date` | `date`, `timestamp with time zone`, `timestamp without time zone` | field type |
| `json` | `jsonb`, `json` | field type |
| primary key (`id`) | `uuid`, `text`, `integer` | field named `id` (name-based; PKs are not in the `Relationships` array) |
| foreign key columns | `uuid`, `text`, `integer` | `Relationships` array from `supabase.ts` (authoritative; name-suffix `_id` is not used) |

The two detection mechanisms are complementary: name-based catches PKs; `Relationships` array catches FKs. They do not overlap.

### Gate condition

**Immediate fail (any count):** Any ❌ or ⚠️ on a primary key column (`id`), any FK column (detected via the generated `Relationships` array, not name-suffix heuristic), or a junction table name or its FK columns. These mismatches indicate a structural problem that A2 cannot safely proceed past.

**Count threshold fail:** More than 3 ❌ + ⚠️ findings on non-key fields across all types combined.

**If gate fails:** Do not start A2. Determine the correct fix:
- If the engine definition has a field the DB lacks → write a new Supabase migration to add the column, apply it explicitly, then re-run A0.
- If the engine definition has a wrong type → correct the engine type definition in `src/cms/types/*.ts`.
- If the engine has a field the DB genuinely does not need → remove it from the engine definition.

## A2 — CMS Engine Integration

### Step 1 — Rebase onto post-A1 develop

```bash
cd .worktrees/cms-engine
git fetch origin
git rebase origin/develop
```

**Verified conflict surface** (from `git diff --stat develop...feature/cms-engine`):

The engine branch touches 69 files. The expected conflicts with A1's changes are concentrated in:
- `src/lib/content-fetchers.ts` — heavily modified by both branches
- `src/lib/supabase-untyped.ts` — small change on engine branch; A1 may restructure it
- Layout components (`professional-case-study-layout.tsx`, `professional-industry-detail-layout.tsx`) — UI changes on engine branch; unlikely to conflict with A1 but verify

The `src/lib/supabase-server.ts`, `src/lib/supabase-browser.ts`, and auth files are **not** in the engine branch's diff — conflict risk there is lower than originally estimated.

**Resolution rule:** Keep A1's security architecture. If a file conflicts between A1's version and the engine's version, preserve A1's module boundary, ESLint rules, and `internal-queries.ts` / `public-query.ts` imports. Adapt engine code to the A1 shape.

### Step 2 — Audit operations layer for A1 ESLint compliance

A1 introduces an ESLint `no-restricted-imports` rule: only files on a defined allow-list may import `internal-queries.ts`. Every file in `src/cms/operations/` that makes Supabase `.from()` calls must be classified using this decision tree:

```
Is this file called from a public route (no auth required)?
  YES → Must use publicQuery() from @/lib/public-query
        If the same file is also called from an admin route, split it:
        extract the public-facing logic into a separate module that uses
        publicQuery(), keep the admin logic in the original using fromTable().
        A single file cannot satisfy both constraints simultaneously.
  NO  ↓

Is this file called at build time (generateStaticParams, ISR) or from an admin route group?
  YES → Add to the internal-queries.ts ESLint allow-list explicitly; use fromTable()
  NO  ↓

Is this file called from a server action (admin CRUD)?
  YES → Route via internal-queries.ts fromTable(); add to allow-list
```

The outcome must be: `npm run lint` passes with 0 errors on the rebased branch.

### Step 3 — Update 8 importers off `@/lib/supabase`

These files import from the deprecated `@/lib/supabase` barrel and must be updated:

| File | What it imports | Correct target |
|------|----------------|----------------|
| `src/app/auth/callback/route.ts` | `createServerSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/case-studies/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/audit-log/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/admin/case-studies/trash/page.tsx` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/api/case-studies/permanent-delete/route.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |
| `src/app/api/case-studies/restore/route.ts` | `createServiceRoleSupabaseClient` | `@/lib/supabase-server` |

After updating all 8, confirm `src/lib/supabase.ts` has zero importers remaining (check both quote styles):

```bash
grep -rE "from ['\"]@/lib/supabase['\"]" src/
```

If the grep returns nothing, delete `src/lib/supabase.ts`.

### Step 4 — Resolve `src/types/supabase-new.ts`

Before deleting, diff the two snapshots:

```bash
diff src/types/supabase.ts src/types/supabase-new.ts
```

Known differences (from prior inspection): `supabase-new.ts` has additional fields on the `algorithms` table (`partner_companies`, `quantum_companies`, `quantum_hardware`, `quantum_software`); `supabase.ts` has `deletion_audit_log` and `legacy_tags_backup` tables that `supabase-new.ts` lacks.

Search for any importers of `supabase-new`:

```bash
grep -r "supabase-new" src/ --include="*.ts" --include="*.tsx"
```

For each importer: if it references one of the fields only in `supabase-new.ts`, verify the field actually exists in the live DB (it should, since `supabase.ts` was just regenerated in A0). Then update the import to `supabase.ts` and delete `supabase-new.ts`.

If the live-regenerated `supabase.ts` is missing a field that `supabase-new.ts` has, that is an A0 gate finding — stop and resolve it before proceeding.

### Step 5 — Verify old admin components were replaced by the engine

The engine branch migrates `industries`, `personas`, and `blog-posts` by replacing the content of their existing admin files in-place — not by deleting and re-creating them. There are no files to manually delete; the rebase brings the replacements automatically.

After rebase, verify the replacement was successful for each of the three types. Note: the blog-posts content type slug is `blog-posts` but its admin route directory is `blog/` (confirmed):

```bash
grep -l "from '@/cms/operations'" \
  src/app/admin/industries/[id]/actions.ts \
  src/app/admin/personas/[id]/actions.ts \
  src/app/admin/blog/[id]/actions.ts
```

All three files must appear in the output. If any are missing, the rebase did not apply the engine migration correctly for that type — resolve the conflict manually.

The file structure for all three types after rebase:

```
src/app/admin/<type>/
  page.tsx          — listing page (retained, unchanged)
  client.tsx        — listing client component (retained, unchanged)
  [id]/
    page.tsx        — detail/edit page (replaced by engine version)
    actions.ts      — server actions (replaced — now imports from @/cms/operations)
    client.tsx      — form client component (replaced by engine version)
```

No additional files (e.g. `new/page.tsx`) exist for these three types — confirmed from inspection.

### Step 6 — Smoke test trash/restore lifecycle

Before the PR is opened, verify the soft-delete lifecycle works end-to-end on the rebased branch:

1. In the admin UI (dev server): soft-delete a case study → confirm it disappears from the public listing.
2. Navigate to `/admin/case-studies/trash` → confirm the deleted record appears.
3. Restore it → confirm it reappears in the admin listing (still as draft/unpublished).
4. Permanent-delete a different record → confirm it is gone from trash.

These routes (`trash/`, `restore/`, `permanent-delete/`) are outside the engine's CRUD scope and are intentionally preserved as-is. The smoke test confirms the rebase did not silently break them.

## Risks

| Risk | Mitigation |
|------|-----------|
| A1 ESLint boundary requires significant refactor of operations layer | Classify all operations files using the decision tree in Step 2 before writing any code; scope the work first. If any file serves both public and admin callers, the split-file rule applies — budget extra time per such file |
| Rebase conflicts larger than the verified conflict surface | Concentrate review on `content-fetchers.ts` and `supabase-untyped.ts`; keep A1's architecture |
| Schema mismatches > threshold found in A0 | Pause A2; determine if fix is a new DB migration or an engine type correction; re-run A0 after fixing |
| A0 finds field in engine that DB lacks → fix requires a new migration, not an engine edit | The gate-fail resolution path explicitly covers this: write the migration, apply it, re-run A0 |
| `supabase-new.ts` has importers relying on fields missing from `supabase.ts` | Diff before deleting; cross-check against live-regenerated types; stop if discrepancy found |
| A0 results go stale if A1 slips > ~2 weeks | Re-run the cross-check script before starting A2 if the gap exceeds 2 weeks |
