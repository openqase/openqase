## Your job — 3 steps

### Step 1: Production SQL verification (read-only)

Open the Supabase SQL editor on the **production** project and run these three queries. They are SELECT-only — no risk.

**Check 1 — audit log policy name:**
```sql
SELECT policyname FROM pg_policies
WHERE tablename = 'deletion_audit_log';
```
Expected: a policy named `"Authenticated users can read audit log"` (or similar) exists.
The migration does `DROP POLICY IF EXISTS "Authenticated users can read audit log"` — confirm the name matches before applying. If the name differs, note it and update the migration file before proceeding.

**Check 2 — anon/authenticated write grants (should still exist — you're checking pre-migration state):**
```sql
SELECT grantee, privilege_type, table_name
FROM information_schema.role_table_grants
WHERE table_schema = 'public'
  AND grantee IN ('anon', 'authenticated')
  AND privilege_type IN ('INSERT', 'UPDATE', 'DELETE', 'TRUNCATE')
ORDER BY table_name, grantee, privilege_type;
```
This shows what the REVOKE migration will remove. Review the list — if any table genuinely needs anon/authenticated writes, it will need a re-GRANT in a follow-up migration (unlikely, but verify).

**Check 3 — `setup_admin_role` grants:**
```sql
SELECT grantee, privilege_type
FROM information_schema.routine_privileges
WHERE routine_name = 'setup_admin_role'
  AND grantee IN ('PUBLIC', 'anon', 'authenticated');
```
Expected: rows exist (confirming the REVOKE will have something to act on). If 0 rows, the migration is a no-op — fine to proceed, the security goal is already met.

If all three checks look reasonable, proceed.
